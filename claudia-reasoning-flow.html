📨 Mensagem do Usuário
Uma nova mensagem chegou via Telegram.
Texto bruto recebido. Ainda sem contexto, intenção ou tipo definidos. O pipeline começa aqui.
fase 1 — classificador (3B)
🧭 Classificar Intenção
Qual é a natureza desta mensagem?
Que tipo de tarefa o usuário quer executar?
Modelo leve (Qwen2.5:3b) analisa a mensagem e produz:
• Categoria: conversa / código / busca / comando / criação
• Nível de complexidade: baixo / médio / alto
• Precisa de ferramentas externas? sim / não
• Rota para: modelo principal ou modelo coder
A mensagem está clara e tem intenção identificável?
NÃO
🔁 Pedir Clareza
Gerar pergunta direta para o usuário.
"Você quer dizer X ou Y?"
O agente não deve adivinhar. Se a intenção for ambígua, é mais eficiente perguntar do que produzir uma resposta errada e depois corrigir.
volta ao início
↩ loop de esclarecimento
SIM
🔀 Rotear para Modelo
Decisão de rota baseada na categoria:
→ Código → Coder model
→ Resto → Modelo principal
Evitar carregar dois modelos ao mesmo tempo por limitação de RAM. A rota determina qual modelo será ativado via Ollama.
FASE 2 — PLANEJAMENTO
📋 sistema de prompt interno — planejamento
"Antes de responder, liste mentalmente os passos necessários para resolver isso.
Para cada passo, identifique: (1) é um raciocínio puro? (2) requer busca/internet? (3) requer execução de comando?
Só então comece a resolver."
planejamento
🗺️ Decompor em Passos
Quais são os passos para resolver isso?
Quais passos dependem de informação externa?
Qual a ordem lógica de execução?
O agente deve listar explicitamente antes de agir:
1. "Para responder X preciso saber Y"
2. "Y está no meu conhecimento ou preciso buscar?"
3. "Se precisar buscar, qual ferramenta?"
4. "Se precisar executar, qual é o risco?"
Preciso de informação ou ação externa para responder isso?
SIM — precisa de ferramenta
FASE 3A — EXECUÇÃO COM FERRAMENTA
execução
🔍 Planejar Uso de Ferramenta
É busca na internet ou comando no terminal?
Qual é o risco desta ação?
Posso reverter se der errado?
Antes de executar qualquer ação:
• Definir o objetivo exato da busca/comando
• Estimar o que esperar como resultado
• Identificar se é destrutivo ou reversível
• Se destrutivo → confirmar com usuário primeiro
Busca web ou comando de terminal?
WEB
🌐 Buscar na Web
Formular query precisa.
Executar busca.
Ler resultado.
Query deve ser específica, não vaga. Se o resultado não responde, reformular antes de tentar novamente. Máximo 3 tentativas.
TERMINAL
💻 Executar Comando
Escrever o comando.
Confirmar se destrutivo.
Executar e ler saída.
Se o comando pode deletar/sobrescrever dados: exibir para o usuário e pedir confirmação explícita antes de executar.
📤 O resultado obtido faz sentido?
O que recebi é o que eu esperava?
Tenho o suficiente para responder?
Ou preciso executar mais um passo?
Se o resultado for vazio, erro ou irrelevante:
→ Reformular a abordagem e tentar de novo
→ Se falhar 3x: informar ao usuário a limitação
NÃO — só raciocínio
FASE 3B — RACIOCÍNIO DIRETO
raciocínio
🧠 Raciocinar Passo a Passo
Pensar em voz alta antes de responder.
Dividir o problema em partes menores.
Resolver cada parte em ordem.
Técnica chain-of-thought aplicada internamente:
1. Reescrever o problema com próprias palavras
2. Identificar o que é dado vs o que precisa ser derivado
3. Resolver da parte mais simples para a mais complexa
4. Verificar se a resposta final é coerente com o input
perguntas guia do raciocínio
  • O que exatamente o usuário quer saber?
  • Existe ambiguidade que eu já devia ter perguntado?
  • Qual o conhecimento mínimo para responder isso?
  • Existem múltiplas interpretações válidas?
  • Minha resposta seria útil para um leigo? E para um especialista?
FASE 4 — VALIDAÇÃO DA RESPOSTA
🧪 sistema de prompt interno — autoavaliação
"Antes de enviar, responda internamente: O que eu produzi responde exatamente o que foi pedido?
Se eu fosse o usuário, ficaria satisfeito com essa resposta? Tem algo óbvio que está faltando?"
autoavaliação
✅ Checar a Própria Resposta
O que produzi responde o que foi pedido?
Há informações erradas ou inventadas?
A resposta está no formato adequado?
Checklist interno antes de enviar:
☐ Responde a pergunta original?
☐ Não tem alucinação ou dado inventado?
☐ Tom e formato adequados ao usuário?
☐ Está conciso ou tem blá-blá desnecessário?
☐ Se usou ferramenta: o resultado foi usado de fato?
O resultado é o que o usuário espera receber?
NÃO — resposta incompleta
🔄 Diagnosticar o Problema
O que está faltando ou errado?
É um problema de planejamento ou execução?
Vale tentar de novo ou admitir limitação?
Se for planejamento falho → volta para Fase 2
Se faltou dado externo → volta para Fase 3A
Se o raciocínio foi incompleto → volta para Fase 3B
Se falhou 2+ vezes → admitir limitação ao usuário
↩ loop de refinamento (máx. 2x)
SIM — aprovado
FASE 5 — RESPOSTA
resposta final
📬 Formatar e Enviar
Adaptar tom e formato para o contexto.
Markdown? Código? Texto simples?
Enviar via Telegram ao usuário.
Regras de formatação:
• Telegram suporta Markdown básico
• Código → bloco com linguagem
• Listas longas → separar em mensagens
• Sempre terminar com algo acionável se possível
🎯 Aguardar Feedback
O usuário ficou satisfeito?
Houve pedido de correção ou continuação?
Se o usuário pedir correção: volta para Fase 1 com novo contexto incluindo a resposta anterior e o feedback.
O contexto da conversa é preservado no histórico de mensagens.
📌 resumo dos prompts de sistema por fase
  • Classificador: "Categorize esta mensagem. Retorne: tipo, complexidade, precisa_ferramenta (bool), rota (main|coder)."
  • Planejamento: "Liste os passos para resolver. Para cada um: é raciocínio, busca ou comando?"
  • Execução: "Antes de executar: defina o objetivo, estime o resultado esperado, avalie o risco."
  • Validação: "Sua resposta resolve o problema original? É precisa? Está no formato certo?"
  • Resposta: "Formate adequadamente para Telegram. Tom direto, sem enrolação."
